Week 10 Notes

MDN: Validating Forms

Client side validation is for user experience. Server side validation is for security.

Built in form validation

Great performance, easy to use, but not customizable.

Html form controls

  • required
  • minlength and maxlength (for strings)
  • min and max (for numerical inputs)
  • type
  • pattern (regex)

CSS pseudoclasses for form validation:

  • :valid
  • :invalid
  • :out-of-range

Form Validation with JavaScript

Constraint Validation API: methods and properties you can use for validation

https://developer.mozilla.org/en-US/docs/Web/API/Constraint_validation

Properties for form element interfaces:

  • validationMessage: a message describing what validation constraints the control doesn't satisfy
  • validity: returns ValidityState object with properties about the validity state of the element

checkValidity() returns true if there is not validity problems

setCustomValidity(message) add a custom error message

Custom error messages are must common use of constraint validation api.

novalidate attribute prevents the form from showing default error messages

MDN: USing Fetch

Headers: for performing actions on the request

response types: basic, cors, opaque. Indicate where theresource has come from and how to treat the response

basic: came from same origin, no restrictions.

cors: came from another origin and has cors headers.

opage: came from a different origin, no cors headers. Can't check if the request was successful.

You can define a mode for fetch requests, so that only certain types will resolve.

Body is used for post requests. Set method and body parameters.
